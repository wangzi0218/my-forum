# PM Workflow Harness

一个面向产品经理个人和小团队的工作流 harness。

它的目标不是再做一个万能 AI 聊天工具，而是帮产品经理把日常工作里的想法、反馈、需求草稿和项目背景，组织成一个能持续推进、能复盘、能记住上下文的产品判断过程。

## 为什么开这个新项目

`PM Method Agent` 已经证明了一个方向：

- 模糊草稿可以被接住。
- 多轮补充可以被承接。
- 阶段关口可以挡住过早方案化。
- 项目背景和记忆建议需要可见、可确认。
- 运行时需要有状态、事件和账本，而不能只靠一段 prompt。

但 `PM Method Agent` 更像一个探索型内核和参考实现。这个新项目会重新收窄范围，优先做一个更简洁、更贴近产品经理真实工作流的工具。

## 一句话定位

`PM Workflow Harness` 帮产品经理把零散想法和团队上下文，变成可以持续推进的产品判断工作流。

它会回答这些问题：

- 这个想法到底在解决什么问题？
- 现在缺的是背景、证据、角色、目标，还是决策？
- 这件事应该继续验证、先暂缓，还是进入方案设计？
- 这条项目背景以后要不要记住？
- 下次继续讨论时，系统应该接着哪里往下看？

## 核心设计原则

我们采用：

```text
harness-first architecture, human-like interaction
```

也就是：

- 底层用明确的 `workspace / case / memory / loop / event` 保证可控。
- 体验层像一个自然的产品同事，少让用户填表，少讲系统术语。
- LLM 负责理解和表达增强，runtime 负责状态、边界和关键决策。
- 记忆必须可见、可确认、可撤回，不做静默长期记忆。

## 第一版先做什么

第一版只追一个端到端闭环：

1. 用户选择或创建一个工作区。
2. 用户输入一句真实草稿、反馈或指标异常。
3. 系统判断这是新问题、继续补充、项目背景，还是查看下一步。
4. 系统输出一张轻卡片。
5. 用户继续自然补充。
6. 系统更新 case 状态，并只追最关键的下一步。
7. 如果出现稳定背景，系统询问是否记住。

## 第一版不做什么

- 不做完整 PRD 自动生成。
- 不做复杂团队后台。
- 不做大型项目管理系统。
- 不做无限聊天机器人。
- 不做复杂 sub-agent 编排。
- 不默认接入所有 IDE、MCP、网页和 CLI。
- 不把所有用户输入都写成长久记忆。

## 和 PM Method Agent 的关系

`PM Method Agent` 是参考实现和方法来源。

这个项目会吸收它的经验，但不会照搬它的复杂度。

保留：

- 阶段关口
- 多轮承接
- 可见记忆
- 运行时账本
- 方案先行拦截

收窄：

- 更少入口
- 更少文档
- 更轻的网页体验
- 更明确的第一版用户路径
