# 验收测试用例

这些测试用例用于验证第一版的核心功能。每个用例有明确的输入、期望行为和通过标准。

---

## TC-01：基础群聊交互

**场景**：用户输入一句草稿，NPC 开始讨论

**前置条件**：应用已启动，用户在默认 PM 讨论场景中

**输入**：
```
运营想在新用户进首页时加一个引导浮层，提升首日发帖率。
```

**期望行为**：
1. 系统识别为"方案先行"场景
2. 小林先发言，追问问题本质
3. 老陈接话，补充事实或追问数据
4. 阿哲可能收敛或给出选项
5. NPC 的发言按顺序出现，不是同时出现

**通过标准**：
- [ ] 小林至少问了一个"为什么"或"你确认过吗"类的问题
- [ ] 老陈至少提到了数据、案例或可行性相关的内容
- [ ] NPC 发言有先后节奏，间隔可见
- [ ] 讨论没有直接进入方案设计

## TC-02：方案先行拦截

**场景**：用户直接提方案，NPC 先质疑问题

**前置条件**：应用已启动

**输入**：
```
我觉得我们应该做一个社区功能。
```

**期望行为**：
1. NPC 不直接讨论怎么做社区
2. 小林追问用户需求的依据
3. 老陈提出风险或成本
4. 阿哲可能直接说"现在不是时候"

**通过标准**：
- [ ] 没有任何 NPC 直接讨论社区功能的实现方案
- [ ] 至少一个 NPC 质疑了"做社区"这个前提
- [ ] 讨论停留在"要不要做"层面，没有进入"怎么做"

---

## TC-03：分歧选择点

**场景**：NPC 讨论出现分歧，生成选择卡片

**前置条件**：用户已输入一个问题，NPC 讨论中出现不同意见

**输入**：
```
最近新用户 7 日留存下降了，运营怀疑是首页内容不够吸引人。
```

**期望行为**：
1. NPC 讨论中出现至少两种不同方向
2. 系统生成选择卡片
3. 卡片显示各选项及 NPC 倾向
4. 用户可以点击选择

**通过标准**：
- [ ] 选择卡片正确显示
- [ ] 卡片包含至少 2 个选项
- [ ] 每个选项有清晰的描述
- [ ] 显示了 NPC 的倾向（谁倾向哪个选项）
- [ ] 用户点击选项后，选择被记录
- [ ] 讨论根据选择继续推进

## TC-04：图片输入

**场景**：用户上传图片，NPC 能识别并讨论

**前置条件**：应用已启动，用户有一张竞品截图

**输入**：
```
[上传一张竞品首页截图]
你看这个产品的首页设计。
```

**期望行为**：
1. 图片在聊天气泡中正确显示
2. NPC 能识别图片内容
3. NPC 在讨论中引用图片内容

**通过标准**：
- [ ] 图片正确显示在消息气泡中
- [ ] NPC 的回复中提到了图片的具体内容
- [ ] NPC 没有给出空泛的回复（如"我看到了图片"）

---

## TC-05：多轮讨论承接

**场景**：用户在同一话题下多轮补充，NPC 能承接上下文

**前置条件**：用户已输入第一个问题，NPC 已讨论一轮

**第一轮输入**：
```
最近有几个客户又提了导出报表的需求。
```

**第二轮输入**：
```
主要是 ABC 公司和 XYZ 公司，他们想要按月导出销售数据。
```

**期望行为**：
1. 第二轮输入被识别为"继续补充"而非"新问题"
2. NPC 承接第一轮的讨论上下文
3. NPC 根据新信息调整讨论方向

**通过标准**：
- [ ] NPC 没有重新问"什么需求"
- [ ] NPC 的回复引用了第一轮的信息
- [ ] 讨论自然推进，没有重置

## TC-06：NPC 说不

**场景**：用户在 YY，NPC 直接反驳

**前置条件**：应用已启动

**输入**：
```
我觉得我们做一个社区功能，用户肯定很喜欢，这样留存就上去了。
```

**期望行为**：
1. 小林质疑"用户肯定很喜欢"的依据
2. 老陈提出社区功能的风险或成本
3. 阿哲可能直接说"现在不是好时机"

**通过标准**：
- [ ] 至少一个 NPC 明确表达了不赞同
- [ ] 没有 NPC 直接附和用户的判断
- [ ] NPC 的反驳有具体理由，不是空泛的否定
- [ ] 用户感觉被质疑了，而不是被支持了

## TC-07：LLM 配置

**场景**：用户配置自己的 LLM API

**前置条件**：应用已启动，进入设置页面

**操作步骤**：
1. 选择提供商（OpenAI / Claude）
2. 填写 API 地址
3. 填写 API Key
4. 填写模型名称
5. 点击"测试连接"

**通过标准**：
- [ ] 提供商切换后，表单正确更新
- [ ] API 地址支持自定义（不是只能用官方地址）
- [ ] 测试连接成功时给出成功提示
- [ ] 测试连接失败时给出具体错误信息
- [ ] 配置保存后，新的 API 被正确使用

---

## TC-08：本地存储

**场景**：讨论记录正确保存和加载

**前置条件**：用户已进行了一次讨论

**操作步骤**：
1. 进行一次完整讨论（至少 3 轮）
2. 关闭应用
3. 重新打开应用

**通过标准**：
- [ ] 讨论记录完整保留
- [ ] 消息顺序正确
- [ ] 选择点的用户选择被保留
- [ ] 图片显示正常

---

## TC-09：JSON 导出

**场景**：用户导出讨论记录为 JSON

**前置条件**：用户已有讨论记录

**操作步骤**：
1. 点击导出按钮
2. 选择保存位置

**通过标准**：
- [ ] 导出的 JSON 文件格式正确
- [ ] JSON 包含完整的讨论记录
- [ ] JSON 可以被重新导入
- [ ] 导入后讨论记录完整

## TC-10：讨论收敛

**场景**：讨论进行多轮后，NPC 给出结论或下一步建议

**前置条件**：用户输入了一个问题，NPC 讨论了多轮

**输入**：
```
门店店员说核销流程太麻烦，经常忙起来就漏掉。
```

**期望行为**：
1. 讨论进行 3-5 轮后，阿哲开始收敛
2. 阿哲总结讨论要点
3. 阿哲给出明确的下一步建议

**通过标准**：
- [ ] 讨论没有无限发散
- [ ] 阿哲在合理时机给出收敛
- [ ] 收敛内容包含具体建议，不是空泛的总结
- [ ] 用户知道下一步该做什么

## TC-11：群聊 UI 体验

**场景**：界面是群聊 IM，足够轻

**前置条件**：应用已启动

**检查项**：

**通过标准**：
- [ ] 界面看起来像群聊，不像管理后台
- [ ] 3 个 NPC 有明确的颜色区分
- [ ] 消息气泡有 NPC 标识（头像或颜色）
- [ ] 没有显示方法论名称
- [ ] 没有显示阶段状态
- [ ] 没有系统术语
- [ ] 输入框简洁，只有文字输入和图片上传
- [ ] 整体界面轻量，不压迫

---

## TC-12：图片粘贴和拖拽

**场景**：用户通过粘贴或拖拽上传图片

**前置条件**：应用已启动，用户有图片在剪贴板或文件系统

**操作步骤**：
1. 复制一张图片到剪贴板
2. 在输入框中 Ctrl/Cmd+V 粘贴

**通过标准**：
- [ ] 粘贴的图片正确显示在输入区域
- [ ] 发送后图片在消息气泡中显示
- [ ] 拖拽图片到聊天区域也能正确上传
- [ ] 图片文件被保存到本地

## 测试执行说明

### 执行顺序

建议按以下顺序执行：
1. TC-11（UI 体验）— 先确认界面基础没问题
2. TC-07（LLM 配置）— 配置好 API 才能测试功能
3. TC-01（基础交互）— 验证核心流程
4. TC-06（NPC 说不）— 验证核心价值
5. TC-02（方案拦截）— 验证关键能力
6. TC-03（选择点）— 验证交互设计
7. TC-04（图片输入）— 验证多媒体支持
8. TC-05（多轮承接）— 验证上下文能力
9. TC-10（讨论收敛）— 验证讨论质量
10. TC-08（本地存储）— 验证数据持久化
11. TC-09（JSON 导出）— 验证备份功能
12. TC-12（图片粘贴）— 验证输入方式
