# 路线图

## 阶段 0：立项准备

目标：把产品范围收窄，避免把旧项目的复杂度整体搬过来。

交付物：

- 产品 Brief
- 架构原则
- 运行原则
- 第一批真实样本

完成标准：

- 能清楚说明这个新项目解决什么、不解决什么。
- 能说清它和 `PM Method Agent` 的关系。
- 能确定第一版只追一个端到端闭环。

## 阶段 1：最小可运行原型

目标：跑通一句草稿到一张卡片的闭环。

建议交付：

- 本地存储
- workspace / case 对象
- 最小 turn loop
- 轻卡片输出
- 5 个真实样本回归

完成标准：

- 用户输入一句真实草稿，系统能判断当前最该补什么。
- 第二轮补充能承接同一个 case。
- 明显方案先行时，系统能先停一下。

## 阶段 2：记忆闭环

目标：让项目背景和用户偏好能被看见、确认和复用。

建议交付：

- memory suggestion
- project profile
- user preference
- “不是这个项目”动作
- 新背景覆盖旧背景

完成标准：

- 用户不用重复解释项目背景。
- 系统不会静默写长期记忆。
- 记忆冲突时，新输入优先。

## 阶段 3：工作台体验

目标：做一个轻量工作台，而不是完整后台。

建议交付：

- 当前案例
- 最近案例
- 当前卡片
- 历史补充
- 记忆建议

完成标准：

- 产品经理能直接用真实草稿试。
- 不需要先理解内部对象。
- 界面不压迫，不像管理后台。

## 阶段 4：团队与入口扩展

目标：在核心闭环稳定后，再考虑团队和入口。

候选方向：

- skill / agent 入口
- MCP 适配
- issue / ticket 集成
- 团队规则
- 轻量分享

暂不承诺：

- 完整权限体系
- 多 agent 编排
- 完整 PRD 生成
- 复杂项目管理
