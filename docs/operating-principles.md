# 运行原则

这份文档记录这个产品的产品和工程底线。

## 1. NPC 必须会说不

这是这个产品存在的核心理由。

当用户直接说"做个弹窗"、"加个入口"、"生成方案"时，NPC 应该先判断问题是否成立。

如果证据不够，要敢于停下来。如果方向不对，要敢于反驳。

说不的方式不是生硬拒绝，而是用提问和反证把用户拉回来。

## 2. 讨论过程必须可见

用户看到的不是"AI 给了你一个答案"，而是"3 个同事在讨论"。

NPC 之间的讨论、分歧、互相回应，全部对用户可见。

## 3. 分歧时选择权交给用户

当 NPC 之间有不同意见时，不能悄悄替用户做决定。

必须：
- 明确告诉用户有哪些选项
- 说明每个 NPC 的倾向
- 让用户自己选择

## 4. 不追求一次问全

NPC 每轮只追最影响推进的一两个问题。

产品经理不是在填问卷。讨论应该是自然的，不是流水线。

## 5. 不把 LLM 当运行时

LLM 可以理解语义、整理表达、生成回复。

但这些事情不应该交给 LLM：

- 决定谁发言（引擎决定）
- 决定是否触发选择点（引擎决定）
- 决定讨论是否该收敛（引擎决定）
- 替用户做最终选择（用户决定）

## 6. 界面必须足够轻

界面就是一个群聊 IM。

不显示：
- 方法论名称
- 阶段状态
- 系统术语
- 复杂的状态面板

用户只需要看消息、做选择、偶尔打字补充。

## 7. 支持图片输入

产品经理日常大量信息通过图片传递：竞品截图、数据报表、设计稿、流程图。

NPC 需要能看懂图片，并在讨论中引用图片内容。

图片通过 base64 传递给多模态 LLM。

## 8. 本地优先

所有数据本地存储，不默认上云。

用户可以导出 JSON 备份，可以自己用 Git 管理版本。

同步是后续功能，不拖慢第一版。

## 9. 每次讨论都要推进

NPC 的每次发言都要让用户知道：
- 我现在怎么理解这个问题
- 还缺什么信息
- 下一步最值得做什么

如果输出只是"看起来有道理"，那就没有帮用户推进工作。

## 10. 不急着做完整平台

第一版先证明一个核心体验：3 个 NPC 的群聊讨论能让产品经理做出更好的判断。

不要一开始就做：
- 登录注册
- 团队权限
- 复杂后台
- 多场景编排
- 大量集成入口
