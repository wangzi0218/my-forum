# 产品 Brief

## 背景

产品经理每天会处理大量不完整的信息：用户抱怨、老板想法、客户反馈、运营指标、竞品观察、会议结论。

这些信息一开始往往不是清晰需求，而是一些还没有被整理好的片段：

- “客户又提了这个功能。”
- “这个指标最近不太好。”
- “老板觉得可以加个入口。”
- “运营想做个新手引导。”
- “这个问题好像要处理，但我还没想清楚。”

真正耗费时间的，不是把它们写成文档，而是判断：这到底是不是问题、值不值得做、下一步该补什么。

## 目标用户

第一版优先面向：

- 独立负责需求判断的产品经理
- 需要沉淀团队产品判断方法的小团队负责人
- 经常和 AI 协作，但不想每次重新写提示词的人
- 希望把项目背景、稳定约束和历史判断保存下来的人

暂不优先面向：

- 大型企业流程审批团队
- 只想自动生成完整 PRD 的用户
- 需要复杂项目排期和资源管理的团队

## 要解决的问题

第一版只解决三个问题。

### 1. 模糊输入怎么被接住

用户不应该先整理完整需求，才能使用系统。

系统应该能接住一句真实草稿，并判断现在最缺什么。

### 2. 上下文怎么被持续承接

用户不应该每次都重复解释项目背景、角色、约束和偏好。

系统需要知道当前工作区、当前案例、已经补过的信息和可复用记忆。

### 3. 什么时候该停下来

系统不应该永远顺着用户往方案走。

当问题还没成立、证据太弱、优先级不够、或者明显可以先暂缓时，系统要能停下来。

## 核心价值

对个人：

- 少重复解释背景。
- 更快知道下一步该补什么。
- 避免太早写方案。
- 保留自己的产品判断过程。

对小团队：

- 统一判断口径。
- 沉淀项目背景和稳定约束。
- 让需求讨论更容易复盘。
- 让新人更容易理解为什么某些需求不该马上做。

## 第一版产品形态

建议第一版是一个轻量工作台。

它可以先是本地网页，也可以后续接入 skill，但第一版体验应该像这样：

1. 用户进入一个工作区。
2. 输入一句草稿。
3. 系统给一张轻卡片。
4. 用户继续自然补充。
5. 系统承接上下文。
6. 系统在必要时询问是否记住项目背景。

不需要一开始做复杂后台。

## 第一版核心对象

### Workspace

一个项目、产品线、团队或个人空间。

保存：项目背景、稳定约束、常用指标、用户偏好、最近案例。

### Case

一个想法、反馈、需求草稿或产品问题。

保存：原始输入、多轮补充、当前阶段、当前卡点、下一步建议、结论或暂缓原因。

### Memory

可复用背景。

区分：项目背景、用户偏好、团队规则、当前案例临时事实。

### Loop

每轮输入如何被承接。

区分：新问题、继续补充、更新背景、查看历史、询问下一步、暂缓或继续。

### Card

用户每轮看到的轻量输出。

回答：我怎么理解、卡在哪里、下一步补什么、是否应该先停。

## 第一版成功标准

第一版不看生成了多少内容，而看这些结果：

- 用户愿意把真实草稿丢进来。
- 用户能看懂系统为什么先问这个问题。
- 用户不用重复补同一段背景。
- 系统能挡住明显过早的方案。
- 系统能在“不急、不值得做”时进入暂缓。
- 输出像一个懂产品的同事，而不是像审查报告。
